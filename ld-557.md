## 项目主要需求

### 一、项目概述

FinanceAPI 是一个面向个人投资者的金融投资分析平台后端 API 服务。系统提供投资组合管理、持仓跟踪、交易记录、市场数据查询和投资复盘等核心功能。采用 NestJS + TypeORM 构建，PostgreSQL 存储数据，Redis 提供缓存加速，JWT 实现认证授权。本项目为纯后端 API 服务，无前端界面。

### 二、核心实体设计

#### 1. 投资组合 Portfolio

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| userId | INT FK | 所属用户 |
| name | VARCHAR | 组合名称 |
| description | TEXT | 组合描述 |
| type | ENUM | 类型（股票/基金/债券/混合/加密） |
| riskLevel | ENUM | 风险等级（保守/稳健/激进） |
| totalValue | DECIMAL | 总市值（计算字段，由持仓汇总） |
| createdAt | TIMESTAMP | 创建时间 |

**关系**：一个用户有多个投资组合，一个投资组合有多个持仓
**业务动作**：创建组合 → 编辑信息 → 添加持仓 → 查看收益 → 删除组合

#### 2. 持仓 Holding

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| portfolioId | INT FK | 所属组合 |
| symbol | VARCHAR | 资产代码（如 AAPL, BTC） |
| quantity | DECIMAL | 持有数量 |
| avgCost | DECIMAL | 平均成本价 |
| currentPrice | DECIMAL | 当前价格（关联 MarketData） |
| pnl | DECIMAL | 盈亏金额（计算字段） |

**关系**：持仓属于某个投资组合，一个持仓有多条交易记录

#### 3. 交易记录 Transaction

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| holdingId | INT FK | 关联持仓 |
| type | ENUM | 交易类型（买入/卖出/分红） |
| quantity | DECIMAL | 交易数量 |
| price | DECIMAL | 交易价格 |
| fee | DECIMAL | 手续费 |
| executedAt | TIMESTAMP | 执行时间 |

**关系**：交易记录属于某个持仓
**业务动作**：买入建仓 → 加仓 → 减仓 → 清仓 → 分红入账

#### 4. 市场数据 MarketData

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| symbol | VARCHAR | 资产代码 |
| name | VARCHAR | 资产名称 |
| price | DECIMAL | 当前价格 |
| change | DECIMAL | 价格变动 |
| changePercent | DECIMAL | 变动百分比 |
| volume | BIGINT | 成交量 |
| marketCap | BIGINT | 市值 |
| updatedAt | TIMESTAMP | 更新时间 |

**业务动作**：查询行情 → 更新价格 → 历史 K 线查询 → 关键字搜索资产

#### 5. 复盘日志 ReviewLog

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| portfolioId | INT FK | 所属组合 |
| period | VARCHAR | 复盘周期（如 2024-Q1, 2024-01） |
| summary | TEXT | 复盘总结 |
| decisions | JSONB | 关键决策记录数组 |
| lessons | TEXT | 经验教训 |
| createdAt | TIMESTAMP | 创建时间 |

**关系**：复盘日志属于某个投资组合
**业务动作**：创建复盘 → 记录决策 → 总结教训 → 按时间查看

### 四、API 路由设计

#### 1. `/api/auth` 认证模块

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | /api/auth/register | 用户注册 | 公开 |
| POST | /api/auth/login | 用户登录，返回 JWT | 公开 |
| POST | /api/auth/refresh | 刷新 access token | 需认证 |
| GET | /api/auth/profile | 获取当前用户信息 | 需认证 |

#### 2. `/api/portfolios` 投资组合

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/portfolios | 获取当前用户的所有组合 | 需认证 |
| GET | /api/portfolios/:id | 获取组合详情（含持仓汇总） | 需认证+所有者 |
| POST | /api/portfolios | 创建投资组合 | 需认证 |
| PUT | /api/portfolios/:id | 编辑投资组合 | 需认证+所有者 |
| DELETE | /api/portfolios/:id | 删除投资组合（级联删除持仓和交易） | 需认证+所有者 |
| GET | /api/portfolios/:id/performance | 组合收益统计（日/周/月/年） | 需认证+所有者 |

#### 3. `/api/holdings` 持仓管理

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/portfolios/:portfolioId/holdings | 获取组合下所有持仓 | 需认证+所有者 |
| GET | /api/holdings/:id | 获取持仓详情（含实时估值） | 需认证+所有者 |
| POST | /api/portfolios/:portfolioId/holdings | 添加持仓（通过交易自动创建） | 需认证+所有者 |
| DELETE | /api/holdings/:id | 删除持仓 | 需认证+所有者 |

**实时估值**：持仓查询时自动关联 MarketData 获取当前价格，计算 pnl = (currentPrice - avgCost) * quantity

#### 4. `/api/transactions` 交易记录

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/holdings/:holdingId/transactions | 获取持仓的交易历史 | 需认证+所有者 |
| POST | /api/holdings/:holdingId/transactions | 记录交易（买入/卖出/分红），自动更新持仓 | 需认证+所有者 |
| GET | /api/portfolios/:portfolioId/transactions | 获取组合全部交易记录（分页+筛选） | 需认证+所有者 |

**交易后自动更新**：买入 → 增加持仓数量并重新计算均价；卖出 → 减少持仓数量；分红 → 记录分红金额

#### 5. `/api/market` 市场数据

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/market/quote/:symbol | 查询单个资产行情 | 需认证 |
| GET | /api/market/search?q=keyword | 搜索资产（按代码/名称模糊匹配） | 需认证 |
| GET | /api/market/history/:symbol | 获取历史 K 线数据（支持 period 参数） | 需认证 |
| GET | /api/market/trending | 获取热门资产排行 | 需认证 |

**缓存策略**：行情数据缓存到 Redis（TTL 60秒），历史数据缓存（TTL 5分钟），搜索结果缓存（TTL 10分钟）

### 五、横切关注点

#### 1. 认证授权 + RBAC

**三级权限体系**：
- **USER**：基础用户，管理自己的投资组合和持仓
- **PREMIUM**：高级用户，额外支持更多组合数量上限、高级分析功能
- **ADMIN**：管理员，可管理所有用户、查看系统统计数据、管理市场数据

**触达层面**：
- **DB**：users 表存储角色和权限信息
- **中间件**：
  - `auth.middleware.ts`：验证 JWT token，注入用户信息到 request
  - `roles.guard.ts`（NestJS Guard）：校验用户角色
- **路由守卫**：使用 NestJS 的 `@UseGuards()` 装饰器在 controller 层控制访问
- **数据范围**：所有组合/持仓查询自动注入 `WHERE userId = currentUser.id` 条件，ADMIN 可绕过此限制

**JWT 结构**：
```json
{
  "sub": 1,
  "email": "user@example.com",
  "role": "USER",
  "iat": 1234567890,
  "exp": 1234571490
}
```

#### 2. 操作日志 + 审计

**必须记录的操作**：
- 所有交易操作：买入、卖出、分红（记录交易详情、操作前后持仓变化）
- 资金变动：组合总值重大变化
- 敏感操作：删除组合、删除持仓

**触达层面**：
- **DB**：`audit_logs` 表
  | 字段 | 说明 |
  |------|------|
  | id | 主键 |
  | userId | 操作用户 |
  | action | 操作类型（CREATE/UPDATE/DELETE） |
  | target | 操作对象（portfolio/holding/transaction） |
  | targetId | 对象 ID |
  | oldValue | 操作前值（JSON） |
  | newValue | 操作后值（JSON） |
  | ip | 请求 IP |
  | userAgent | 请求 UA |
  | createdAt | 操作时间 |

- **中间件**：`audit.middleware.ts`（NestJS Interceptor）自动拦截指定路由的写操作并记录
- **Service 层**：`audit.service.ts` 提供 `log()` 方法，供 service 在业务逻辑中手动调用记录

### 六、技术实现要求

- **框架**：NestJS，使用 Module/Controller/Service/Entity 分层架构
- **ORM**：TypeORM，entity 使用装饰器定义
- **数据库**：PostgreSQL 15，提供 migration 和 seed 文件
- **缓存**：Redis，使用 `cache-manager` + `cache-manager-redis-yet` 集成
- **认证**：JWT（`@nestjs/jwt`），密码使用 bcrypt 加密
- **校验**：class-validator + class-transformer，DTO 层统一参数校验
- **API 文档**：Swagger（`@nestjs/swagger`），自动生成 OpenAPI 文档，访问路径 `/api/docs`

### 七、全局异常处理

**后端统一异常处理**：
- **异常过滤器** `http-exception.filter.ts`：捕获所有 HttpException，统一返回格式
- **全局异常过滤器** `all-exceptions.filter.ts`：捕获所有未处理异常，记录错误日志，返回通用 500 错误
- **统一响应格式**：
```json
{
  "statusCode": 200,
  "message": "success",
  "data": {},
  "timestamp": "2024-01-01T00:00:00.000Z",
  "path": "/api/portfolios"
}
```
- **错误响应格式**：
```json
{
  "statusCode": 400,
  "message": ["name should not be empty"],
  "error": "Bad Request",
  "timestamp": "2024-01-01T00:00:00.000Z",
  "path": "/api/portfolios"
}
```

**限流**：使用 `@nestjs/throttler` 实现 API 限流，基础用户 60次/分钟，高级用户 120次/分钟

## 部署要求

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

docker-compose.yml 要求：
- **不要写 `version:` 字段**
- 顶层声明 `name: ldapi`
- 所有服务使用 `container_name` 命名容器，带 `${COMPOSE_PROJECT_NAME:-ldapi}` 前缀
- 数据库和 Redis 数据使用命名卷持久化
- 后端配置 `healthcheck`（检查 `/api/health` 端点），`depends_on` 数据库和 Redis 健康
- 数据库配置 `healthcheck`（pg_isready）
- Redis 配置 `healthcheck`（redis-cli ping）
- 所有可变配置通过环境变量注入
- 后端服务暴露端口 `38505`
- 无需前端容器（纯后端项目）

需要提供：docker-compose.yml, .env, .env.example, backend/Dockerfile
